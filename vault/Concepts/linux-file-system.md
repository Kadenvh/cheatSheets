# Linux File System

## Quick Reference
- `/` : Root of everything // All paths descend from here
- `/etc` : System configuration files // network, users, services
- `/var` : Variable data // logs (/var/log), mail, spool, temp state
- `/proc` : Virtual filesystem for processes // /proc/PID/ exposes process state, /proc/cpuinfo for CPU
- `/dev` : Device files // /dev/sda (disk), /dev/null (discard), /dev/urandom (random)
- `/home` : User home directories // Each user gets /home/username
- `/tmp` : Temporary files // Cleared on reboot (or by tmpwatch)
- `/usr` : User programs and libraries // /usr/bin, /usr/lib, /usr/share
- `/opt` : Optional third-party software // Self-contained installs
- `stat file` : Show inode, size, timestamps, permissions // Everything stored in the inode
- `ln source link` : Hard link // Same inode, same data — deleting one keeps the other
- `ln -s target link` : Symbolic link // Pointer to a path — breaks if target moves
- `mount /dev/sda1 /mnt` : Attach filesystem // `umount /mnt` to detach
- `lsblk` : List block devices // Shows disks, partitions, mount points
- `findmnt` : Show mount tree // Cleaner than parsing /etc/fstab

## Functional Logic
- **Concept:** [[Linux File System]] // Hierarchical tree where everything — files, devices, processes — appears as a file.
- **Dependency:** [[Linux CLI Essentials]] // Navigation and file operations require understanding the tree structure.
- **FHS Standard:** Filesystem Hierarchy Standard defines where things go // Predictable across distros — /etc is always config, /var is always state.
- **Inodes:** Metadata structure on disk — permissions, timestamps, block pointers, NOT the filename // Filenames are just directory entries pointing to inodes.
- **Hard Links:** Multiple directory entries → same inode // Data persists until ALL hard links are removed. Cannot cross filesystems.
- **Soft Links:** File containing a path string // Flexible (cross-filesystem, link to dirs) but fragile (dangling if target removed).
- **Virtual Filesystems:** /proc and /sys are generated by the kernel at runtime // No disk backing — reading /proc/meminfo queries kernel directly.
- **Mount Points:** Attaching a filesystem to a directory in the tree // Root `/` is always mounted first; other devices mount on top.
- **Permissions Inheritance:** New files get permissions from umask, not parent dir // `umask 022` → files get 644, dirs get 755.

## Implementation
```bash
# Show inode number and link count for a file
ls -li /etc/hostname
# 131073 -rw-r--r-- 1 root root 12 Mar 15 hostname
# col 1: inode | col 3: link count

# Create a hard link and verify same inode
echo "data" > original.txt
ln original.txt hardlink.txt
stat original.txt hardlink.txt | grep Inode
# Both show same inode number, link count = 2

# Create a soft link and demonstrate breakage
ln -s /tmp/target.txt /tmp/symlink.txt
rm /tmp/target.txt
ls -l /tmp/symlink.txt  # Shows broken link (red in ls --color)

# Find largest directories under /var
du -sh /var/*/ 2>/dev/null | sort -rh | head -5

# Check filesystem type and usage
df -Th  # -T shows filesystem type (ext4, xfs, tmpfs)
```

## Sandbox
- Map out the top-level FHS directories on your system and identify which are on separate partitions
- Create a hard link and a soft link, then delete the original — observe which survives
- Use /proc/self/ to discover your shell's PID, file descriptors, and environment variables
